# MarkAtlas (마크아틀라스) - Global Trademark Explorer

React 기반에서 다국가 상표 데이터 통합 / 검색 UX / 확장 가능한 아키텍처 설계를
종합적으로 경험하기 위해 제작한 프로젝트입니다.
MSW 기반으로 서버 상황을 사전에 검증하며, 실제 서비스 연동을 전제로 한 구조로 설계했습니다.

## 🌐 배포 주소

(Vercel 배포 후 주소 추가)

## 🎯 프로젝트 목적과 의도

1. 다국가 API 데이터 스키마를 공통 포맷으로 통합하는 설계 경험
2. 서버 상태 / 클라이언트 상태를 분리한 상태 관리 전략 수립
3. 무한스크롤·필터링 기반 검색 중심 UX 설계
4. 실제 서비스 확장을 고려한 구조 (API, 인증, 상태 관리)

## 🚀 주요 기능

### ✔ 핵심 기능

- 다국가 상표 데이터 조회
- 국가별 상이한 API 응답을 공통 데이터 포맷으로 변환
- 조건 기반 검색 및 필터링(등록 상태, 출원번호, 출원일자, 언어 정렬 ㄱ-ㅎ / A-Z)
- 즐겨찾기 기능
- 항목 상세 정보 모달

### ✔ UX / 고급 기능

- IntersectionObserver 기반 무한스크롤
- 로딩 / 에러 / Empty State 처리
- 무한스크롤 시 MoveTop 버튼 노출
- 반응형 웹 지원

## 🧱 기술 스택

- React
- Vite
- Tailwind CSS v4
- React Query (Server State)
- Axios
- MSW (Mock Service Worker)
- Context API (Client State)

## 📁 폴더 구조

src/
┣ api/ # API 요청 로직
┣ components/ # 도메인별 UI 컴포넌트
┣ hooks/ # 비즈니스 로직 커스텀 훅
┣ contexts/ # 전역 클라이언트 상태
┣ utils/ # 공통 유틸 / 데이터 스키마 변환
┣ App.tsx
┗ main.tsx

- 도메인 기준으로 폴더를 구성하고 비즈니스 로직과 UI로직을 분리했습니다.
- 컴포넌트는 **도메인을 기준으로 분리**하였고 공통으로 사용되는 UI는 'common' 컴포넌트로 분리했습니다.

## 🔧 주요 로직 소개

### ✔ 다국가 데이터 스키마 통합

- API 응답 직후 국가별 데이터를 공통 포맷으로 변환
- 이후 필터링 / 즐겨찾기 / UI 렌더링은 국가 구분 없이 동일한 데이터 구조 기준으로 동작

### ✔ 상태 관리 전략

- 서버 상태: React Query
  - 캐싱 / 로딩 / 에러 처리 구조 선제 구성
  - 실제 서버 연동 시 최소 수정으로 확장 가능

- 클라이언트 상태: Context API
  - 즐겨찾기, 모달 상태 등 전역 접근이 필요한 UI 상태만 관리

- 로컬 상태: useState
  - 검색 조건, 선택 아이템 등 컴포넌트 내부 상태

### ✔ 무한스크롤 설계

- IntersectionObserver 기반
- 초기 마운트 시 observer 미동작 이슈 해결
- 로직을 useInfiniteScroll 커스텀 훅으로 분리하여 observer 책임/상태 업데이트 흐름 명확화/재사용성 확보

## 💡 배운 점 & 인사이트

- 다국가 데이터를 다루는 경우 초기 스키마 설계가 전체 복잡도를 결정
- UI 컴포넌트에서 분기 처리하는 방식은 확장성에 치명적
- 서버 상태를 React Query로 분리하니 로딩/에러 처리 구조가 매우 안정적
- 무한스크롤은 “기능”보다 타이밍과 책임 분리가 핵심
