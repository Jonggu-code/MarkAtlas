# ⚙️ 사용 기술

- React
- Tailwind CSS (v4)
- React Query
- Axios
- MSW (Mock Service Worker)
- Vite

# ✅ 주요 기능

- MSW 활용 환경 구성 (실제 서버 연동 고려)
- 다국가 데이터 조회 (국가별 스키마 차이를 공통 포맷으로 통합하여 처리)
- 조건 기반 검색 및 필터링 (등록 상태, 출원번호, 출원일자별, 언어 순서별 (ㄱ-ㅎ / A-Z))
- 즐겨찾기 기능
- IntersectionObserver 기반 무한스크롤
- 각 항목 상세정보 모달창
- 로딩 및 에러 처리, 기본 화면 (아이템 없을 시) 구현
- 무한스크롤로 인한 MoveTop 버튼 표시
- 반응형 웹사이트

# 🛠️ 폴더/컴포넌트 구조

src/
┣ api/ # API 요청 로직
┣ components/ # 도메인별 UI 컴포넌트
┣ hooks/ # 비즈니스 로직을 분리한 커스텀 훅
┣ contexts/ # 전역 클라이언트 상태
┣ utils/ # 순수 유틸 함수

- 도메인 기준으로 폴더를 구성하고 비즈니스 로직과 UI로직을 분리했습니다.
- 컴포넌트는 **도메인을 기준으로 분리**하였고 공통으로 사용되는 UI는 'common' 컴포넌트로 분리했습니다.

# 📖 사용 기술 설명

## 상태 관리 전략 & 이유

- **서버 상태는 React Query를 사용**해 관리했습니다.
- 현재 프로젝트는 MSW를 사용한 로컬 환경에서 동작하지만, 추후 실제 서버와의 연동을 고려하여 최소한의 수정으로 확장 가능하도록 React Query를 채택하였습니다.
- 이를 통해 **API 데이터 캐싱, 로딩 및 에러 처리 구조**를 미리 구성하였습니다.

- **전역 클라이언트 상태는 Context API를 사용**했습니다.
- 각 목록의 즐겨찾기 상태, 알림 모달창과 같이 여러 컴포넌트에서 접근해야 하지만 복잡한 상태 구조나 미들웨어가 필요하지 않은 경우에 한정하여 사용했습니다.

- **컴포넌트 로컬 상태는 useState를 사용**했습니다.
- 목록 필터링 조건, 선택된 아이템 구분(모달창) 등 컴포넌트 내부에서만 사용되는 상태에 한하여 적용했습니다.

## 다국가 데이터 처리 설계

- API 응답을 받은 직후, 공통 데이터 포맷으로 변환하여 사용했습니다.
- 이후 모든 로직(필터링, 즐겨찾기, UI렌더링)은 국가 구분 없이 동일한 데이터 구조를 기준으로 동작하도록 설계했습니다.

## 공통 컴포넌트 구성 방식

- 컴포넌트 재사용성과 책임 분리를 위해 common 폴더를 만들어 비즈니스 로직을 포함하지 않는 순수 UI 컴포넌트만 배치하였습니다.

# 📌 트러블 슈팅

## 다국가 데이터 스키마 통합 설계

- 프로젝트 구현 초기에는 UI 컴포넌트에서 국가별 분기 처리를 진행하려고 했으나, 국가별로 다른 UI를 구성하거나, 추후 다른 국가가 추가되면 유지보수가 힘들것이라 판단했습니다.
- API 응답 직후 데이터를 공통 포맷으로 변환하고, 로직에서는 국가 구분 없이 동일한 데이터 구조를 사용하도록 설계했습니다.(utils/dataSchema.js)
- 그 결과, UI와 비즈니스 로직의 복잡도를 줄였고 추후 국가 추가에도 유연하게 대응할 수 있는 구조를 확보했습니다.

## 무한스크롤 초기 마운트 문제 해결

- IntersectionObserver를 활용한 무한스크롤 구현 과정에서 초기 마운트 시 observer가 동작하지 않았습니다.
- DOM 렌더 타이밍과 observer 등록 시점의 차이로 인해 발생했다고 판단하였고, 로직을 커스텀 훅(hooks/useInfiniteScroll.js)으로 구성해 observer 에 대한 책임과 상태 업데이트 흐름을 분리하였습니다.
- 그 결과, 초기 로딩부터 안정적으로 동작하는 무한스크롤 구조를 구현하고, 재사용성을 확보했습니다.

## 컴포넌트 책임 분리 및 구조 개선

- 프로젝트 초기 App이 담당하던 데이터 조회, 필터링, 무한스크롤 등의 역할을 개선하기 위해 로직을 커스텀훅으로 분리했습니다.
- 그 결과, 컴포넌트 간 책임이 명확해지고 유지보수가 용이한 구조로 변화하였습니다.

# 개선하고 싶은 부분

## 전체적인 UI/UX 개선

- 현재 UI는 핵심 기능에 집중할 수 있도록 검색 영역(SearchForm)과 아이템 목록(ItemList)의 가시성을 중심으로 UI를 구성했습니다.
- 추후 필터 조건이나 기능이 확장될 경우, 추가된 요소들을 함께 고려하여 사용자가 기능을 직관적으로 인지할 수 있는 방향으로 UI/UX를 개선하면 좋을 것 같습니다.

## 로그인 기능 구현

- 현재 프로젝트는 인증 기능 없이 진행했지만, 향후 로그인 기능을 확장할 수 있는 구조로 설계했습니다.
- 향후 토큰 기반 인증 방식을 가정하고, 인증 상태는 전역 클라이언트 상태로 관리하여 서버 상태와 명확히 분리할 계획입니다.
- 이를 통해 로그인 여부에 따른 UI 분기와 사용자별 기능 접근 제어를 적용하고, 개인화 기능을 확장하면 좋을 것 같습니다.
